SQL Injection

At the end of the day, web apps are just tools that allow end users to interact with data in complicated ways. Developers streamline the process as much as possible behind the scenes, but there is a limit to what they can do. It usually comes to a point where users need to do some work, and give the code some information to work with. This information is often manipulated and filtered through other bits of code, and winds up as an input for a database query. But allowing users unfettered access to the database can be dangerous. Without the necessary filters, malicious users can manipulate the code in an unintended way, causing it to spit out the wrong data.

For example, lets say you allow the user to interact with the database with minimal security precautions. If a user forgets their password, they will likely be prompted by a box to plug in their username and email address. Behind the scenes, the code might look something like this:

username_input = input; sql_query = "SELECT password FROM db WHERE username = " + username_input

If everything goes as planned, users who forget their passwords will merely type in their username, and the system will work fine. Now, let's say someone wants to be an asshole; in the user_input field they type: " xxx OR 1=1". This means that the query will turn into this:

sql_query = "SELECT password FROM db WHERE username = xxx OR 1=1"

The xxx is unimportant, but the 'OR 1=1' is. It will ALWAYS be true for every row in the db (because 1 always equals 1), which means that it will return passwords for all rows.

What to do

Okay, so how do we fix this? One easy way is to add a bit of nuance to the SQL query. Instead of manipulating the query directly, you can tell it that it should interpret username_input literally. In SQL-talk, you can do this using the @ symbol:

sql_query = "SELECT password FROM db WHERE username = @1"; sql_query.execute(username_input)

Now, username_input is passed to the query as an argument instead of a string to be added onto the end.
